# Project Architecture

This repository contains a **NixOS Flake** that manages two machine‑level configurations (``ms-02`` and ``n10-nixos``) and two macOS/Home‑Manager configurations (``macbook-pro`` and ``mac-mini``). The goal is to keep infrastructure as code, share common modules, and make future host additions straightforward.

## Top‑level layout
```text
├─ flake.nix                    # Flake entry point – bootstraps system & home configs
├─ flake.lock                   # Pinned input revisions (keep in sync with CI)
├─ opencode.json                # OpenCode agent configuration
├─ README.md                    # High‑level instructions
├─ modules/                      # Reusable NixOS modules
│   ├─ system/                 # System‑level (kernel, network, services etc.)
│   │  ├─ core/
│   │  ├─ networking/
│   │  ├─ hardware/
│   │  ├─ virtualization/
│   │  ├─ services/
│   │  └─ security/...
│   └─ user/                   # User‑level shared fragments (scripting, shell)
├─ hosts/                        # Host‑specific configuration
│   ├─ ms-02/
│   │  ├─ configuration.nix
│   │  └─ hardware-configuration.nix
│   ├─ n10-nixos/
│   ├─ macbook-pro/
│   │  └─ home.nix
│   └─ mac-mini/
│       └─ home.nix
├─ home/                         # Shared Home‑Manager (user) modules
│   └─ indlns.nix                # Common Home‑Manager fragments for user *indlns*
└─ secrets/                       # SOPS encrypted data (no secrets committed)
    ├─ common.yaml
    └─ .sops.yaml
```

## Core concepts

| Concept | What it does | Where you see it |
|---|---|---|
| **Modules** | Small, composable Nix expressions that add or tweak system properties. They are imported from `modules/system/` (system‑wide) and merged into host configs via the `mkHost` helper in `flake.nix`. | `modules/**/`, `flake.nix`
| **Hosts** | Each physical or virtual machine has a dedicated folder under `hosts/` with a `configuration.nix`. The autogenerated `hardware-configuration.nix` should never be edited. | `hosts/<host>/configuration.nix`
| **Home‑Manager** | Per‑user configuration lives in `${HOME}/home.nix`, imported by the Home‑Manager module for macOS hosts. All user tooling (CLI, shell, git) is defined under `home/` or `modules/user/`. | `hosts/<mac>/home.nix`, `home/indlns.nix`
| **Secrets** | Managed with sops‑nix; the repository stores only encrypted data and config references. No plain secrets are committed. | `secrets/common.yaml`, `modules/system/security/sops.nix`

## How to add a new host

1. Create a folder under `hosts/` (e.g., ``my-new-host``).
2. Run `nixos-generate-config --root /mnt/my-new-host` to create the autogenerated `hardware-configuration.nix`.
3. Write a minimal `configuration.nix` that imports your system modules and sets the desired options (often just point at existing modules via `imports = [ ./path/to/configuration.nix ]`).
4. In `flake.nix`, add a line inside ``nixosConfigurations``:
   ```nixosConfigurations.my-new-host = mkHost "my-new-host" ./hosts/my-new-host/configuration.nix;``` 
5. Test locally: `sudo nixos-rebuild switch --flake .#my-new-host`.

## How to add a new system module

1. Add a file under `modules/system/`, e.g., ``modules/system/services/custom/foo.nix``.
2. Export the module as a simple function with optional parameters.
3. In the host or `defaults.nix`, optionally import it by adding an entry to the `imports` list or using the conditional `lib.optional`. For example, to enable the new service on all hosts:
   ```
   ++ lib.optional (hostName != "") ./services/custom/foo.nix
   ```
4. Test the module in a sandboxed rebuild (`sudo nixos-rebuild test`).

## How to edit Home‑Manager fragments

All user‑specific fragment files (e.g., `home/indlns.nix`, or anything under `modules/user/`) are imported automatically into each macOS host via the `home-manager` module. When adding a new setting:
1. Edit the relevant fragment file.
2. Run `sudo nixos-rebuild switch --flake .#macbook-pro` (or `mac-mini`).
3. Verify that the change took effect: e.g., `ls ~/.zshrc` for shell changes, or open Vim to confirm plugin installation.

## Secrets management workflow

1. Add a new secret by running:
   ```bash
   sops --encrypt secrets/common.yaml > secrets/<new>.yaml.sops
   ```
2. Reference it in modules using `config.sops.secrets.<name>`.  The `home` module automatically wires user‑specific variables.
3. Run `nix flake check .` to ensure all referenced keys exist and the config builds successfully.
