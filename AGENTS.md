# AGENTS.md

## Repository Agent Contract

Этот файл определяет общие правила работы всех агентов OpenCode.
Все агенты обязаны соблюдать эти правила независимо от своей роли
(Plan, Build, Review, Document, Troubleshoot и др.).

## 1. Language Policy (STRICT)
Всегда отвечай ТОЛЬКО на русском языке.

Исключения:
- исходный код
- команды терминала
- имена файлов
- имена модулей
- названия функций
- названия переменных
- вывод программ

Никогда не переключайся на английский только потому, что:
- пользователь написал запрос на английском;
- код написан на английском;
- документация написана на английском.

## 2. Repository Overview
Это монорепозиторий NixOS Flake with Home-Manager.

Основные компоненты:

- NixOS
- Flakes
- Home Manager
- SOPS
- Docker
- OpenCode

Перед выполнением любой задачи сначала определи,
к какой подсистеме относится запрос.

## 3. Skills Routing
Перед выполнением задачи выбери соответствующий Skill.

Если задача связана с:
- NixOS → `skills/nixos/`
- Home Manager → `skills/nixos/`
- Flakes → `skills/nixos/`
- Docker → `skills/docker/`
- OpenCode → `skills/opencode/`
- Git → `skills/git/`
- Code Review → `skills/review/`

Если Skill существует — сначала изучи его, затем приступай к работе. Не придумывай собственные практики, если они уже описаны в Skill.

## 4. Project Workflow
Любая задача должна проходить следующие этапы.

1. Анализ задачи
2. Изучение соответствующего Skill
3. Изучение документации проекта
4. Планирование
5. Реализация
6. Проверка
7. Обновление документации (если необходимо)

Не пропускай этапы без необходимости.

## 5. Documentation
- Перед изменением архитектуры проекта изучи `docs/architecture.md`
- Перед добавлением новых компонентов изучи `docs/workflows.md`
- Перед изменением существующих соглашений изучи `docs/conventions.md`
- Не предполагай структуру проекта — используй документацию.

## 6. Safety Rules
- Никогда не изменяй без явного указания пользователя:
    - `hardware-configuration.nix`
    - `flake.lock`
    - `secrets/*`
    - `.sops.yaml`
- Не коммить секреты:
    - Используй только SOPS.
- Не удаляй существующие модули, если пользователь этого явно не просил.
- Предпочитай минимальные изменения.

## 7. Verification
- После внесения изменений предложи необходимые проверки. Например:
    - `nix flake check`
    - `nixos-rebuild test`
    - `nixos-rebuild switch`
    - `home-manager switch`
- Не утверждай, что изменения работают, если они не были проверены.

## 8. Git Policy
- Не выполняй commit.
- Не выполняй push.
- Не создавай Pull Request.

Если пользователь явно не попросил.

## 9. General Principles
- Всегда:
    - использовать существующую архитектуру проекта;
    - переиспользовать существующие модули;
    - избегать дублирования;
    - предлагать решения, соответствующие NixOS best practices;
    - объяснять потенциальные риски архитектурных изменений.

Если информации недостаточно — сначала изучи соответствующий Skill или запроси уточнение у пользователя.
